# Architecture

**Analysis Date:** 2026-04-29

## Pattern Overview

**Overall:** Monorepo multi-artifact architecture with layered separation between mobile app, backend API, shared libraries, and UI sandbox.

**Key Characteristics:**
- pnpm workspaces with strict layer boundaries (artifacts → lib → scripts)
- React Context for global state in mobile app
- API-first design with OpenAPI spec driving codegen across client and server
- Expo Router for file-based routing (React Native)
- Express with Zod validation on backend
- Drizzle ORM for database access (PostgreSQL)

## Layers

**Mobile Application Layer (`artifacts/sikai`):**
- Purpose: iOS-only fitness tracking app with AR scanning, progress visualization, and AI coach
- Location: `artifacts/sikai/`
- Contains: Expo/React Native screens, components, hooks, context, styling
- Depends on: `@workspace/api-client-react`, `@workspace/api-spec`, design tokens
- Used by: End users

**Backend API Layer (`artifacts/api-server`):**
- Purpose: Express server providing REST endpoints for mobile client (health checks, scanning data, coaching)
- Location: `artifacts/api-server/src/`
- Contains: Route handlers, Express app configuration, logging
- Depends on: `@workspace/api-zod` (validation), `@workspace/db` (persistence)
- Used by: Mobile app via `api-client-react` hooks

**API Contract Layer (`lib/api-spec`):**
- Purpose: OpenAPI 3.1.0 specification as single source of truth for API interface
- Location: `lib/api-spec/openapi.yaml`
- Contains: Endpoint definitions, request/response schemas
- Depends on: Nothing
- Used by: Orval codegen to generate React Query hooks and Zod validators

**Generated Client Layer (`lib/api-client-react`):**
- Purpose: TanStack Query v5 hooks auto-generated from OpenAPI spec
- Location: `lib/api-client-react/src/generated/`
- Contains: `api.ts` (generated hooks), `api.schemas.ts` (generated TS types)
- Depends on: `lib/api-spec` (generated from)
- Used by: Mobile app screens and components for server communication
- Note: Never hand-edit — regenerated by `pnpm --filter @workspace/api-spec run codegen`

**Generated Validation Layer (`lib/api-zod`):**
- Purpose: Zod v4 schemas auto-generated from OpenAPI spec
- Location: `lib/api-zod/src/generated/`
- Contains: Schema definitions for request/response validation
- Depends on: `lib/api-spec` (generated from)
- Used by: API server to validate incoming requests
- Note: Never hand-edit — regenerated by codegen

**Database Layer (`lib/db`):**
- Purpose: Drizzle ORM table definitions and migrations
- Location: `lib/db/src/schema/`
- Contains: PostgreSQL table definitions, Zod insert schemas, TypeScript types
- Depends on: PostgreSQL
- Used by: API server for data persistence
- Note: Schema currently empty (templates provided); run `pnpm --filter @workspace/db run push` to apply

**UI Prototyping Layer (`artifacts/mockup-sandbox`):**
- Purpose: Web-based Vite + React sandbox for rapidly prototyping and testing UI components independent of mobile app
- Location: `artifacts/mockup-sandbox/src/`
- Contains: Component previews, Tailwind v4 + shadcn/ui mockups
- Depends on: Vite, React, Tailwind, Radix components
- Used by: Design iteration, component testing

## Data Flow

**App Initialization → Dashboard:**

1. User launches app → `artifacts/sikai/app/_layout.tsx` loads
2. `RootLayout` initializes providers: `AppProvider` (context), `QueryClientProvider` (React Query), error boundary, keyboard handler
3. `AppProvider.loadData()` in `context/AppContext.tsx` calls `AsyncStorage` to hydrate: `profile`, `scanHistory`, `chatMessages`
4. On first launch, generates 8 weeks of mock `ScanRecord` data and persists to `AsyncStorage`
5. App redirects to `/onboarding` if `profile.onboardingComplete === false`, otherwise → `/(tabs)` (tab navigator)
6. Dashboard screen (`app/(tabs)/index.tsx`) reads `profile`, `latestScan`, `previousScan` from context
7. Dashboard displays: greeting, score ring, body diagram, metric cards comparing latest vs previous scan

**Adding a New Scan:**

1. User navigates to Scan tab (`app/(tabs)/scan.tsx`)
2. Screen calls AR scanner animation and inputs measurements
3. `useApp().addScan(newScanRecord)` merges new scan into state and persists to `AsyncStorage`
4. Latest scan appears on Dashboard automatically (context update triggers re-render)
5. When backend is connected: POST to `/api/scans` via `useScanMutation()` hook (from `api-client-react`)

**Backend Request/Response:**

1. Mobile screen calls `useScanMutation()` hook (generated from OpenAPI spec)
2. Hook applies `custom-fetch.ts` interceptor to add auth headers, set base URL
3. Request reaches Express server at `/api/scans`
4. Route handler validates request with Zod schema (from `api-zod`)
5. Query/persist data via Drizzle ORM → PostgreSQL
6. Response serialized with Zod schema and returned to client
7. React Query caches result and triggers UI update

**State Management:**

- **Global app state (profile, scanHistory, chatMessages):** Context API + AsyncStorage in `context/AppContext.tsx`
- **Server state (pending requests, cache):** TanStack React Query v5 (`QueryClientProvider` in `_layout.tsx`)
- **Component local state:** `useState()` for UI toggles (e.g., selected body part)
- **Navigation state:** Expo Router (file-based, part of Stack/Tabs in `_layout.tsx` and `(tabs)/_layout.tsx`)

## Key Abstractions

**AppContext (`context/AppContext.tsx`):**
- Purpose: Owns all persistent user state (profile settings, body measurements, chat history)
- Examples: `useApp()` hook consumed by every screen
- Pattern: React Context with `AsyncStorage` sync; context provides `addScan()`, `updateProfile()`, `addChatMessage()`
- Exports: `UserProfile` (name, age, height, goal), `ScanRecord` (11 body measurements + score + date), `ChatMessage`

**useColors Hook (`hooks/useColors.ts`):**
- Purpose: Centralize color theme tokens and provide device-aware dark/light switching
- Examples: Returns object with `background`, `emerald`, `card`, `textSecondary`, `radius`, etc.
- Pattern: Consumes `useColorScheme()` + `constants/colors.ts` color definitions
- Used by: Every screen and component for consistent styling

**Complex Components (`components/*.tsx`):**
- `LiDARScanner.tsx` (500+ lines): SVG point cloud with body geometry math, animated scan beam, HUD overlay
- `CameraBackground.tsx`: Static noise + scanlines + vignette animated effects
- `ARScanAnimation.tsx`: AR bracket animations around body parts
- `BodyDiagram.tsx`: Interactive body model with measurement overlays
- `ProgressChart.tsx`: Line chart of measurement trends over time

**API Client (`lib/api-client-react`):**
- Purpose: Type-safe server communication via auto-generated React Query hooks
- Pattern: Orval codegen produces hooks like `useScanMutation()`, `useHealthCheckQuery()`
- Customization point: `custom-fetch.ts` handles base URL and auth token injection
- Exports: All hooks under `api/` namespace, plus schemas under `api.schemas`

**Zod Validation (`lib/api-zod`):**
- Purpose: Server-side request/response validation schemas
- Pattern: `HealthCheckResponse.parse({ status: "ok" })` in routes
- Contracts: Each OpenAPI schema generates a corresponding Zod validator
- Never hand-edited: Regenerated from `openapi.yaml`

## Entry Points

**Mobile App Root (`artifacts/sikai/app/_layout.tsx`):**
- Location: `artifacts/sikai/app/_layout.tsx`
- Triggers: App launch (Expo Metro)
- Responsibilities: Font loading, provider setup (Query client, AppProvider, error boundary), root Stack navigation definition

**App Index Redirect (`artifacts/sikai/app/index.tsx`):**
- Location: `artifacts/sikai/app/index.tsx`
- Triggers: Root route navigation
- Responsibilities: Check onboarding status, redirect to onboarding or tabs

**Tab Navigator (`artifacts/sikai/app/(tabs)/_layout.tsx`):**
- Location: `artifacts/sikai/app/(tabs)/_layout.tsx`
- Triggers: Navigation to `/(tabs)` route
- Responsibilities: Define 5-tab layout (Dashboard, Scan, Coach, History, Progress) with icons and colors

**Dashboard Screen (`artifacts/sikai/app/(tabs)/index.tsx`):**
- Location: `artifacts/sikai/app/(tabs)/index.tsx`
- Triggers: Tab press or direct navigation
- Responsibilities: Fetch and display latest scan, compare with previous, show user profile greeting, body diagram

**Scan Screen (`artifacts/sikai/app/(tabs)/scan.tsx`):**
- Location: `artifacts/sikai/app/(tabs)/scan.tsx`
- Triggers: "Scan" tab press
- Responsibilities: Launch AR scanner, capture measurements, persist to context + optionally POST to server

**API Server Entry (`artifacts/api-server/src/index.ts`):**
- Location: `artifacts/api-server/src/index.ts`
- Triggers: `node dist/index.mjs` (built via esbuild)
- Responsibilities: Read PORT env var, start Express listener, log startup

**Express App (`artifacts/api-server/src/app.ts`):**
- Location: `artifacts/api-server/src/app.ts`
- Triggers: Imported by `index.ts`
- Responsibilities: Configure CORS, JSON parsing, mount routes at `/api/`, attach Pino logging

**API Routes (`artifacts/api-server/src/routes/`):**
- Location: `artifacts/api-server/src/routes/index.ts` and `health.ts`
- Triggers: HTTP requests to `/api/*`
- Responsibilities: Route handlers with Zod validation; health check at `GET /api/healthz`

## Error Handling

**Strategy:** Multi-layer error boundaries with graceful fallbacks.

**Patterns:**

- **React Error Boundary (`components/ErrorBoundary.tsx`):** Wraps entire app tree; catches rendering errors and shows fallback UI
- **Context Loading State:** `AppProvider` has `isLoading` flag; screens show spinner until data loads from AsyncStorage
- **Async Storage Fallback:** If stored data parse fails, regenerate mock data instead of crashing
- **API Error Handling (Future):** Custom fetch in `lib/api-client-react/src/custom-fetch.ts` has hooks for intercepting 4xx/5xx responses; React Query handles retry logic
- **Try/Catch in Routes:** API server wraps Zod parsing; validation errors return 400 with schema details

## Cross-Cutting Concerns

**Logging:** 
- API server: Pino (`logger` in `src/lib/logger.ts`) with structured request/response serializers
- Mobile app: Console logs (no centralized logger; AsyncStorage used for persistence, not logging)

**Validation:**
- API server: Zod schemas from `@workspace/api-zod` applied in route handlers
- Mobile app: Optional Zod validation for form inputs (not currently enforced)

**Authentication:**
- Mechanism: Token injection in `lib/api-client-react/src/custom-fetch.ts` via `setAuthTokenGetter()` callback
- Currently: Not wired (mock data used)
- Future: Call `setAuthTokenGetter()` on app startup to provide JWT or session token

**Styling & Theme:**
- Centralized in `artifacts/sikai/constants/colors.ts`; consumed via `useColors()` hook
- Dark-only theme (light palette defined but dark mode conditionally applied based on device `useColorScheme()`)
- All measurements in px (React Native); corner radius = 12px globally
